# 2026-04-11 — notes.version production schema rollout (B1-3)

## Context

Learning phase B1 step 3 of the backend learning plan. Added a
monotonically increasing `version` column to the `notes` table to set
up future "edit history" / event sourcing features. **This is not an
optimistic lock** — there is no `WHERE version = ?` CAS in any write
path. The full design rationale (including why optimistic locking is
the wrong fit for note content) is documented in
`docs/learn-backend/phase-b1.md` under the B1-3 section.

## Schema change

```sql
ALTER TABLE notes ADD COLUMN version INTEGER NOT NULL DEFAULT 0;
```

Generated by Drizzle at `drizzle/0026_bright_puppet_master.sql`.

## Write paths that increment version

- `notes.update` — every user content write
- `notes.appendBlocks` — Ask AI "append to another note"

## Write paths that do **not** increment version (intentional)

- `notes.enableShare` / `notes.disableShare` — share metadata only, not content
- `folders.delete` — bulk `folderId` reassignment for children on folder removal
- `notes/journal-titles.ts` — system-driven journal title normalization

## Rollout command (production Turso)

```bash
node scripts/db/apply-2026-04-11-notes-version-rollout.mjs
```

The script is idempotent — re-running after the column already exists
is a no-op (prints `SKIP — column already present`).

## Local verification

`pnpm db:push` refused the change because drizzle-kit conservatively
flags `NOT NULL without default` on SQLite ALTER (even though the
schema has `.default(0)`). Applied the same `ALTER TABLE` directly
against `data/second-brain.db` via `@libsql/client` — both local and
production databases now have the column with identical type
(`INTEGER NOT NULL DEFAULT 0`).

## Risks / follow-ups

- No application-level CAS — concurrent multi-tab edits still behave
  as LWW (last-write-wins). Intentional for B1-3; real conflict
  resolution is deferred to B10 (CRDT / Yjs).
- `notes.appendBlocks` previously lacked cache invalidation. Not fixed
  in B1-3 — out of scope. Flagged as a follow-up.
- Rollout is idempotent; safe to re-run.
