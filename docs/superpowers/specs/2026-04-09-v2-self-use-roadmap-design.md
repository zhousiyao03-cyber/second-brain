# Second Brain V2（自用优先）路线图 — 设计文档

## 概述

当前项目已经具备「笔记 / 学习 / 项目阅读 / Ask AI / Focus / Token Usage」等丰富能力，但核心问题不是功能不够，而是**使用阻力过高**：

- 进入页面会卡，切页和首屏体感不顺
- 编辑器功能很多，但输入、阅读、操作的一致性还不够好
- Ask AI 的 UI 和交互摩擦较多，难以成为日常工作流的一部分
- 收集资料之后，用户不愿意继续停留、继续读、继续沉淀

因此，V2 的目标不是继续扩张模块，而是先把产品修成一个**愿意天天打开、愿意停留、愿意继续阅读和思考**的环境。

## 背景判断

结合当前仓库状态和使用反馈，可以把问题分成三层：

### 1. 体感层问题

- `/ask`、`/notes`、dashboard 等高频页面都带有较重的 client 侧逻辑
- 编辑器承载了 Mermaid、Excalidraw、表格、图片布局、块操作等大量高级能力，初始负担偏大
- 页面上的 loading、数据请求、复杂控件挂载容易叠加成"哪里都能用，但哪里都不够顺"

典型关联文件：

- `src/app/(app)/ask/page.tsx`
- `src/app/(app)/notes/page.tsx`
- `src/app/(app)/page.tsx`
- `src/components/editor/tiptap-editor.tsx`

### 2. 工作流层问题

- 资料能收进来，但进入后不容易继续读下去
- Ask AI 仍然更像一个单独页面，而不是当前阅读 / 写作上下文里的低摩擦助手
- 首页更像 dashboard，而不是"今天该继续什么"的工作台

### 3. 知识复利层问题

- 现有 RAG 和内容组织已经有底子，但还没有形成 Karpathy 在 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 里强调的那种"持续编译、持续维护"的知识中间层
- 如果基础体验不顺，知识复利机制很难真正跑起来

## V2 北极星

V2 的北极星不是"功能数量更多"，而是：

> 当用户收集了一批资料后，愿意再次打开 Second Brain，并且能顺畅地继续阅读、继续提问、继续沉淀。

换句话说，V2 首先要解决的是**愿不愿意用**，其次才是**用完能不能复利**。

## 非目标

V2 当前阶段不优先追求：

- 再增加新的独立业务模块
- 优先做多模态、复杂 agent 工作流、自动化编排等重量级能力
- 为了展示效果而做一套很完整但自己未必高频使用的产品包装
- 过早投入大规模知识图谱 / 图形化关系视图

## 路线选择

### 路线 A：性能优先

先解决页面卡顿、编辑器输入延迟、Ask AI 首屏和交互响应问题，让整个产品重新变得"可久留"。

优点：

- 最直接命中当前痛点
- 能立刻改善日常使用意愿
- 为后续阅读体验和知识复利打基础

代价：

- 短期内不会新增很多"看起来很强"的新能力

### 路线 B：阅读体验优先

优先做阅读模式、沉浸式排版、回看入口、渐进展开等能力。

优点：

- 更直接面向"沉下心来读"

代价：

- 如果底层卡顿和交互阻塞仍然存在，体验提升会被稀释

### 路线 C：知识编译优先

优先做 canonical pages、自动摘要回写、知识 lint、长期维护等能力。

优点：

- 最接近 LLM Wiki 的长期形态

代价：

- 在当前体验基础上做，容易出现"系统思想正确，但用户不想打开"的问题

### 推荐决策

推荐采用：

`性能优先 -> 阅读体验优先 -> 知识编译优先`

也就是：

1. 先把卡顿和摩擦修掉
2. 再把编辑器和阅读体验打磨成真正愿意停留的环境
3. 最后把高价值问答和资料处理编译进长期知识结构

## V2 设计原则

### 1. 高频路径优先

优先优化用户最常走的路径，而不是最酷的能力：

`打开应用 -> 找到要继续的内容 -> 阅读 / 写作 -> 提问 -> 沉淀结果`

### 2. 默认界面克制

高级功能继续保留，但不应该在默认界面上持续抢注意力。V2 的默认界面应该更安静、更专注。

### 3. Ask AI 必须融入上下文

AI 不该只是一个单独页面，而应该逐步融入：

- 当前 note
- 当前选区
- 当前 learning topic
- 当前 project

### 4. 首页是工作台，不只是统计页

首页的职责应该是回答：

- 我现在最值得继续什么？
- 哪些资料值得回看？
- 哪些内容刚刚被沉淀？
- 哪些问题还没解决？

### 5. 知识复利建立在低摩擦之上

在用户愿意持续回来之前，不要过度投入复杂知识编译系统。复利系统必须建立在"愿意用"的基础之上。

## 里程碑路线图

### Milestone 1：流畅度修复

### 目标

让高频页面从"能用但卡"变成"明显顺"。

### 范围

- 审查高频页面的 client-heavy 结构和重复请求
- 识别首屏不必要的同步挂载内容
- 将重编辑器能力改成按需加载或延迟挂载
- 清理明显拖慢切页和首屏的 UI 结构
- 对 Ask AI、Notes、Dashboard 做一次专门的摩擦排查

### 重点关注

- `src/app/(app)/ask/page.tsx`
- `src/app/(app)/notes/page.tsx`
- `src/app/(app)/page.tsx`
- `src/components/editor/`

### 成功标准

- 常用页面切换时不再有明显卡顿感
- 编辑器输入和滚动保持稳定
- Ask AI 首屏可以快速进入可交互状态
- 最常见的小摩擦问题有一轮系统清理

### Milestone 2：编辑器重构为"写作 + 阅读环境"

### 目标

让编辑器不只是一个功能丰富的 block editor，而是一个愿意停留的思考环境。

### 范围

- 梳理"写作模式 / 阅读模式 / 结构模式"的职责边界
- 收敛默认可见的 hover 控件、插入控件和块操作噪音
- 优化长文阅读时的宽度、节奏、目录、引用、代码块和图片排版
- 提升键盘优先体验，减少细碎鼠标交互
- 补强"继续上次阅读 / 上次编辑位置"能力

### 成功标准

- 用户在 note 中更容易进入持续阅读状态
- 默认界面更克制，不会被控件不断打断
- 长文内容的阅读舒适度明显提升

### Milestone 3：Ask AI 融入工作流

### 目标

把 Ask AI 从一个独立页面，逐步变成当前工作上下文里的低摩擦助手。

### 范围

- 落地编辑器内 Ask AI 入口
- 支持基于当前 note / 当前选区 / 当前 topic / 当前 project 发问
- `@mention` 或等价机制，用于 pin 特定上下文
- 常见输出动作化：
  - 插入到当前文档
  - 追加到当前文档
  - 生成新 note
  - 提取成摘要 / 问题 / TODO
- 引用来源更轻量，不打断阅读和写作

### 对 `/ask` 页的定位调整

- 保留 `/ask`，但逐步把它定位为"研究台"或"跨知识域问答台"
- 日常简单提问尽量回到具体上下文里完成

### 成功标准

- 用户不需要频繁切到 `/ask` 才能用 AI
- AI 结果更容易沉淀回知识库
- Ask AI 从"聊天页"变成"工作流插件"

### Milestone 4：首页改造为 Today Workspace

### 目标

让首页回答"我现在该继续什么"，而不是只展示统计卡片。

### 建议首页模块

- Continue reading / Continue writing
- 最近值得回看的收藏或项目资料
- 最新被 AI 整理出的摘要或结构化结论
- 未解决问题 / 需要继续推进的问题
- 今日建议回顾的旧笔记

### 成功标准

- 打开应用后，用户能立即找到下一步动作
- 首页把用户重新拉回知识工作流，而不是停留在管理视图

### Milestone 5：知识编译层

### 目标

在基础体验顺滑后，逐步落地 LLM Wiki 风格的持久知识中间层。

### 范围

- 引入 canonical knowledge pages，例如：
  - topic
  - concept
  - thesis
  - project
  - open question
- 新资料 ingest 后，优先更新这些页面，而不只是做一次性摘要
- 高价值 Ask AI 结果可一键沉淀为知识页
- 周期性知识 lint：
  - 矛盾结论
  - 过期内容
  - 缺失反向链接
  - 高频提及但没有独立页的概念

### 成功标准

- 知识不只是"能搜到"，而是会被持续编译和维护
- 高价值结论不再只停留在聊天记录里

## Backlog 分层

### P0：立刻做

- 高频页面性能审计和卡顿定位
- 编辑器重能力拆分与按需加载
- Ask AI 页面 UI 收窄和状态机简化
- 高频小 bug / 小摩擦清扫
- 首页从"数据总览"改成"下一步工作台"

### P1：体感稳定后做

- 阅读模式和继续阅读能力
- 编辑器内 Ask AI
- `@mention` 精确上下文
- AI 输出动作层
- 来源引用轻量化

### P2：体验稳定后做

- canonical knowledge pages
- 知识 lint
- 自动 resurfacing / review
- 更完整的长期知识编译与维护系统

## 成功衡量

V2 是否成功，不看功能数，而看下面这些问题是否变好了：

- 收集资料之后，用户是否更愿意回来继续读
- 打开应用后，是否更快进入实际工作状态
- 编辑器是否更像思考环境，而不是功能拼装体
- Ask AI 是否更少打断、更容易沉淀结果
- 产品是否开始具备长期知识复利，而不是单次问答能力

建议同时记录两组观测信号：

- 产品信号：
  - 每周主动回访 Second Brain 的次数是否上升
  - 收集后的资料在 7 天内被重新打开和继续阅读的比例是否上升
  - Ask AI 的回答被保存回 note 或转成后续行动的比例是否上升
- 工程信号：
  - 高频页面的首屏可交互时间是否明显下降
  - 编辑器输入和滚动是否不再出现明显卡顿
  - `/ask` 页面是否不再出现大量阻塞式等待和状态跳变

## 风险与取舍

### 风险 1：继续堆功能，掩盖真正问题

如果 V2 继续优先上新模块或大能力，可能会让根本性体验问题继续积累。

### 风险 2：只做性能，不改工作流

如果只修卡顿，不重构编辑器和 Ask AI 的工作流位置，用户仍然可能觉得"虽然快了一点，但还是不想待着"。

### 风险 3：过早做知识编译层

如果在基础体验还不顺时投入大量精力做 canonical pages 和自动维护，结果可能是系统变复杂了，但使用频率没有提升。

## 推荐的下一步

基于本设计文档，下一步应输出一份可执行的实施计划，优先覆盖：

1. Milestone 1 的性能与摩擦排查
2. Milestone 2 的编辑器交互收敛
3. Milestone 3 的 Ask AI 工作流融入

这三个方向完成后，再决定是否开启 Milestone 5 的知识编译层建设。
