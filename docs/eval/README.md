# Knosi Eval ‚Äî Quickstart

A repeatable quality measurement for Ask AI (RAG retrieval + end-to-end agent answer). The point: changes to RAG / prompt / model / tools should be measured, not guessed.

## When to run

**Before the first commit of any change to these files**:
- `src/server/ai/agentic-rag.ts`
- `src/server/ai/chat-prepare.ts`
- `src/server/ai/chat-system-prompt.ts`
- `src/server/ai/provider/*.ts`
- `src/server/ai/reranker.ts`
- `src/server/ai/tools/*.ts`
- `src/server/ai/embeddings.ts`
- `eval/ground-truth.json` or `eval/agent-cases.json` (annotation set itself)

**After**: paste the `pnpm eval:compare` output into the commit message body or PR description.

## Five-second usage

```bash
# One-time per session (fetches the prod KNOSI_SECRET_KEY from k3s).
pnpm eval:prepare

# RAG retrieval-layer eval (ground-truth.json, scored by recall@5/10 + MRR).
pnpm eval:rag --user <your-user-id> --top-k 16 --out eval/results/run-<label>.json

# End-to-end agent eval (agent-cases.json, scored by citation/mention/negative).
pnpm eval:agent --user <your-user-id> --out eval/results/agent/run-<label>.json

# Compare a run against the committed baseline.
pnpm eval:compare eval/results/baseline.json eval/results/run-<label>.json
pnpm eval:compare eval/results/agent/baseline.json eval/results/agent/run-<label>.json
```

Find your user id once and stash it:
```bash
EVAL_USER_ID=$(pnpm eval:prepare > /dev/null && \
  cat eval/.eval-env | grep TURSO_AUTH | head -1 > /dev/null; \
  echo "5dcad5a2-1d20-43df-818c-d640958ddb8a")  # or query users table
```

## How to read the output

**RAG aggregate**:
- `Recall@5` / `Recall@10`: of the chunks you labeled relevant, how many were in the retriever's top-5 / top-10
- `MRR`: how high in the ranking is the first relevant chunk (1.0 = first place, 0.5 = second place, 0 = not in top-K)

**Agent aggregate**:
- `Pass rate`: cases where citation = mention = negative = 1.0 simultaneously
- `Citation score`: did the agent's tool calls (or RAG-injected preamble) cover the expected note ids
- `Mention score`: did the answer text contain the required substrings
- `Negative score`: did the answer avoid the forbidden substrings

The three sub-scores are kept independent on purpose. A weighted-sum hides which dimension regressed. Always look at the sub-scores when the pass rate moves.

**Compare output**:
- `Aggregate` block with delta arrows per metric
- `REGRESSIONS` block (agent only): cases that went PASS ‚Üí FAIL ‚Äî investigate before merging
- `FIXES` block (agent only): cases that went FAIL ‚Üí PASS
- `Notable per-case movements`: top per-case Œî ‚Äî read this even when aggregate doesn't move much, individual cases often tell the real story

## Adding a query (RAG layer)

1. Edit `eval/ground-truth.json`, append to `queries[]`:
   ```json
   {
     "id": "q-019",
     "query": "your real query phrasing",
     "category": "free-form tag",
     "relevant_chunk_ids": [],
     "notes": "what made you add this case?"
   }
   ```
2. Run seed mode to see what the retriever surfaces today:
   ```bash
   pnpm eval:rag --user <id> --seed-template
   ```
3. Hand-label `relevant_chunk_ids` from the candidates. Honest labels only ‚Äî wrong labels permanently corrupt the baseline.
4. Re-run baseline and update `eval/results/baseline.json`. Mention the new case in the commit message.

## Adding a case (agent layer)

Edit `eval/agent-cases.json`, append to `cases[]`. Required fields: `id`, `query`, `category`, `expected_tool_touched_note_ids`, `must_mention`, `must_not_mention`, `rubric`. See existing cases for reference.

`expected_tool_touched_note_ids` is the set of note ids the agent must reach (via `searchKnowledge` / `readNote` tool calls **or** via the RAG-injected preamble ‚Äî the harness counts both).

## Anti-patterns

- **"Pass rate dipped ‚Äî must be a regression"** ‚Äî Always check the sub-scores. Citation up + Mention down often means the agent answered correctly but the test substring was too strict. Update the case, don't revert the code.
- **"Add a case to make a number look better"** ‚Äî The case set should reflect real usage, not be tuned to current implementation. Mine real cases from `chat_tasks` (see `scripts/_extract-real-queries.mjs`).
- **"Run eval after I commit"** ‚Äî Eval is to inform the commit, not document it. Run before, paste the result into the commit message.
- **Skipping eval because "it's just a small change"** ‚Äî Small changes to ranking are exactly the kind that move multiple cases by 0.1 each in opposite directions. The aggregate hides it; per-case Œî shows it.

## Architecture notes

- The harnesses dynamic-import production TS files via `tsx` (which resolves `@/` paths). They share `runChatStream` with production `/api/chat` to avoid drift ‚Äî see commit `c69f5f5` for that refactor.
- `eval/.eval-env` is generated by `eval-prepare-env.mjs` from `.env.turso-prod.local` + Hetzner k3s secret. Gitignored. Re-run `eval:prepare` if the prod key rotates.
- Eval runs against **production Turso** (your real notes). It does not modify production state ‚Äî only reads. But it does call DeepSeek (or whatever your prod chat provider is) for the agent eval, costing tokens. Budget ~5‚Äì10 min and a small token bill per full agent run.
